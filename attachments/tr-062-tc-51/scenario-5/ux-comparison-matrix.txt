=== TC-51 Scenario 5 — UX COMPARISON skill vs Obvious.ai ===
Date: 2026-06-10
Contexto: data specialist ejecuta migración Tessitura → veevartd
Sesión observada: Scenario 2 (skill) + Scenario 3 (Obvious.ai)

================================================================================
DIMENSIÓN 1 — CLARIDAD DEL PROGRESO
================================================================================
Sub-pregunta: ¿el specialist sabe en todo momento qué está pasando, qué objeto
se está cargando, cuántos records van?

Skill (dashboard Vite):
  ✅ Banner global con status (STAGED · DRY-RUN → APPROVED · LOADED)
  ✅ KPIs siempre visibles: Total records, Objects, Payloads, Prefix
  ✅ Approval gate con progress bar 0/6 → 6/6 payloads
  ✅ Per-object payload status (loaded/failed/staged)
  ✅ Dashboard live en http://localhost:5177/ paralelo a la conversación
  ✗ Algunas vistas (Failure explorer) sólo se pueblan si hay failures
  ✗ Tiempo total en wall clock no se muestra explícitamente

Obvious.ai (chat-based):
  ✅ Mensajes secuenciales describen cada paso (verify → stage → load → verify)
  ✅ Resumen final por objeto con counts y status
  ✅ Wall clock total reportado (~300s incluyendo retry)
  ✗ No hay vista persistente fuera del chat — para ver progreso anterior hay
    que scrollear
  ✗ Sin dashboard separado, el specialist comparte real estate con el bot

VEREDICTO #1: dashboard skill MEJOR para visibilidad multi-fase + observabilidad
paralela. Obvious.ai EQUIV. para progreso lineal de una migración pequeña.
================================================================================

================================================================================
DIMENSIÓN 2 — SURFACING DE BLOCKERS / DECISIONS
================================================================================
Sub-pregunta: cuando algo requiere decisión del specialist (mapping ambiguo,
schema delta, side effect), ¿la herramienta lo expone con suficiente claridad?

Skill (Decisions/Mapping Deltas section):
  ✅ 7 decisions visualizadas en dashboard con tag (approve/warn) + justificación
     - QA isolation field, Opp.StageName hardcode, Membership dates, Program lookup,
       Payment keying, Trigger bypass, Email Deliverability
  ✅ 7-field write confirmation gate explícito (target/operation/object/rows/payload/
     side-effects/rollback) — pego literal en chat o no hay write
  ✅ NPSP triggers fire flagged como warn antes del load

Obvious.ai (chat-based safety gate):
  ✅ Safety gate ejecutado pre-load con 6 checks (RecordTypes, ExtID fields,
     Membership fields, Address fields, Deliverability, pre-existing records)
  ✅ vnfp__Membership_* absent detectado automáticamente y corregido
  ✅ Rollback plan reportado en línea ("DELETE WHERE *_External_ID__c LIKE 'tess-obvious-%'")
  ✗ No hay confirmación textual obligatoria del operador entre stage y load —
    el flow continúa después del safety gate sin un pause humano explícito

VEREDICTO #2: skill MEJOR por el gate humano obligatorio (7-field confirmation
literal). Obvious.ai es EQUIVALENTE en discovery automático de schema deltas y
rollback planning, pero no fuerza pause humano antes de side effects.
================================================================================

================================================================================
DIMENSIÓN 3 — APROBACIÓN DE MAPPINGS
================================================================================
Sub-pregunta: ¿el specialist puede revisar y aprobar los mappings field-by-field
ANTES del primer write?

Skill (Mapping Viewer):
  ✅ Mapping Viewer mostró 6 entries con External-ID field por objeto + QA pattern
  ✅ Counts staged por objeto visibles
  ✅ Mapping deltas vs spec baseline visibles con razón
  ✗ Visualización field-by-field a nivel de columna NO observada (no fue scope
    detallar las 16 vistas)

Obvious.ai (chat report):
  ✅ Pre-load: lista los 6 objects + ExtID field cada uno + counts staged
  ✅ Mapping inline reportado en respuesta del bot
  ✗ No hay edición interactiva de mapping antes del load — si el specialist quiere
    cambiar un mapping debe hacerlo en lenguaje natural y re-prompt al bot

VEREDICTO #3: skill MEJOR por revisión estructurada en dashboard. Obvious.ai
ES MÁS ÁGIL para mappings standard pero más cumbersome para overrides custom.
================================================================================

================================================================================
DIMENSIÓN 4 — NAVEGACIÓN / DESCUBRIBILIDAD
================================================================================
Sub-pregunta: ¿el specialist puede saltar entre vistas, encontrar lo que necesita?

Skill (16 vistas Vite):
  ✅ Sidebar con vistas dedicadas (Overall, Per-object progress, Upload runs,
     Failure explorer, Mapping viewer, Decisions, …)
  ✅ URL paths permiten compartir vistas (http://localhost:5177/...)
  ✗ 16 vistas para una migración pequeña es overhead — el specialist principiante
    podría perderse

Obvious.ai (linear chat):
  ✅ Una sola superficie (chat) — zero curva de aprendizaje de navegación
  ✗ Encontrar info de un paso anterior requiere scroll/search en chat
  ✗ Workbook "QA Migration — Tucson Botanical" mencionado pero no es navegación
    primaria

VEREDICTO #4: skill MEJOR para migración compleja (multi-batch, multi-object,
debugging). Obvious.ai MEJOR para migración simple end-to-end sin desvíos.
================================================================================

================================================================================
DIMENSIÓN 5 — EXPORTABILIDAD DEL REPORTE FINAL
================================================================================
Sub-pregunta: cuando termina la migración, ¿el specialist puede llevarse el
reporte para handoff, audit, debugging?

Skill (workspace files):
  ✅ Workspace persistente en filesystem (raw/, staging/, mappings/, upload-runs/,
     reports/, status/, dashboard/)
  ✅ Acceptance report archivado con counts + decisions + side effects
  ✅ 6 staging CSVs disponibles para diff campo-a-campo
  ✅ project-field-map.json + mapping-diff.json + status/record-upload-status.json
  ✅ Todo bajo el directory del run → trivial de zippear y compartir

Obvious.ai (workbook artifact):
  ✅ Workbook "QA Migration — Tucson Botanical" creado con 6 staging CSVs
  ✅ Chat persistente sirve como log narrativo
  ✗ Reporte final compuesto por mensajes de chat — no es un PDF/HTML único
  ✗ Compartir requiere recortar respuestas del bot o exportar workbook por separado

VEREDICTO #5: skill MEJOR para handoff técnico (archivos estructurados).
Obvious.ai EQUIV. para handoff narrativo (chat-style).

================================================================================
VENTAJAS Y DESVENTAJAS POR HERRAMIENTA
================================================================================

SKILL (veevart-data-migration-skill)
  Ventajas:
  + Dashboard live observable paralelo a chat
  + Gate humano obligatorio (7-field confirmation) que fuerza pause cognitivo
  + Workspace persistente con archivos estructurados (audit/handoff trivial)
  + Decisions visibles con justificación (transparencia interna)
  + Mapping deltas vs spec baseline → easy debugging cuando spec quede stale
  Desventajas:
  - 16 vistas es overhead para migración trivial
  - Mapping spec puede quedar stale (ej: HomePhone) y la skill lo carga aunque
    el field no exista (Bulk API silent-ignores → defect latente)
  - Requiere Claude Code instalado + Vite dev server local

OBVIOUS.AI
  Ventajas:
  + Zero install/setup en host (todo en web app)
  + Discovery automático en runtime corrige schema deltas (vnfp__Membership_* →
    npe01__) sin requerir update de spec
  + Wall clock total medido explícito
  + Curva de aprendizaje cero (chat-based)
  Desventajas:
  - Sin gate humano obligatorio entre stage y load
  - Sin vista persistente — debug requiere scroll en chat
  - Setup OAuth por org (External Client App) repetitivo
  - Edición de mappings vía re-prompt (no UI estructurada)

================================================================================
VEREDICTO FINAL — ACCIONABLE
================================================================================

Recomendación: CONVIVENCIA con criterio de uso

Adoptar la skill para:
  - Migraciones grandes (>1000 records) donde el dashboard live justifica el overhead
  - Migraciones críticas / production-bound donde el gate humano es feature, no fricción
  - Handoffs técnicos donde el workspace estructurado es requerido
  - Debugging de mappings stale o decisions ambiguas
  - Especialistas que prefieren control granular sobre transparencia interna

Mantener Obvious.ai para:
  - Migraciones rápidas / exploratorias (sub-100 records)
  - Specialists con poca formación técnica (curva cero)
  - Casos donde el discovery automático en runtime es preferible a un spec mantenido
  - Migraciones repetitivas donde el chat-flow es más eficiente

Convivencia:
  - Ambas pueden coexistir en el toolkit del specialist
  - Para certificación final de una migración crítica, usar skill (audit trail)
  - Para POCs / scoping inicial, usar Obvious.ai (rapidez)
  - Diff campo-a-campo entre tools (como este TC-51) sigue siendo válido para
    detectar regresiones / spec drift

DECISIÓN PARA IM-788: la skill puede entregar el mismo resultado funcional que
Obvious.ai con mejor UX en 4 de 5 dimensiones. No reemplaza a Obvious.ai para
todos los casos, pero es ADOPTABLE como herramienta primaria para migraciones
críticas con audit requirements.
