Foco · Triage de Incidentes (ya en producción)

🚨 Triage de Incidentes

Toma un bug, case o incidente reportado y lo convierte en una investigación accionable: qué se sabe, qué repo revisar primero, qué cambio parece sospechoso, qué impacto tiene y qué evidencia falta antes de escalar.
En producción
/incident-triage-agent El Investigador
¿Qué hace? Lee el ticket de Jira y la evidencia disponible, revisa repos locales e historial de git, y arma un reporte de triage con hipótesis, severidad, impacto, ventana probable de introducción y próximos pasos.
Paso a paso (en simple)
  • Lee el bug o case en Jira y resume síntomas, fechas, comportamiento actual y esperado.
  • Clasifica el dominio técnico: frontend, backend, Salesforce, POS, datos, integración o cross-system.
  • Identifica el repositorio probable y alternativas para revisar primero.
  • Busca cambios sospechosos en git, PRs, commits, archivos o módulos relacionados.
  • Separa hechos de hipótesis para evitar culpar personas sin evidencia directa.
Entrega
Un Incident Triage Report con bug, área afectada, severidad, impacto, ventana probable, cambio sospechoso, ownership técnico, análisis y evidencia faltante.
Gate humano Decisión QA
¿Qué decide? QA usa el reporte para confirmar si la hipótesis es suficiente, si falta reproducir el bug, si hay que pedir más evidencia o si el caso puede pasar a formalización/fix.
Criterios de salida
  • Hay un repo probable y archivos o módulos concretos para inspeccionar.
  • La severidad tiene una razón clara de negocio o flujo afectado.
  • La ventana de introducción indica fechas o cambios candidatos.
  • Los datos faltantes quedan explícitos para no asumir comportamiento no probado.
Siguiente paso
Confirmar reproducción o evidencia mínima antes de crear un bug formal, asignar fix o escalar a otro agente del workflow.

Foco · QA Engineer Asistente (ya en producción)

🤖 QA Engineer Asistente

Toma una historia de Jira y la lleva hasta tener el plan, los casos, la ejecución y los defectos levantados. Corre con dos skills que ya están en uso real sobre el proyecto IM.
En producción
/qa-bundle-generator El Planeador
¿Qué hace? Lee una historia de Jira y arma el plan de pruebas completo: lo escribe en el Jira QA Sub-task y crea el archivo de casos en el repo.
Paso a paso (en simple)
  • Lee la historia de Jira, sus criterios de aceptación y ejemplos.
  • Detecta lo que falta documentado y lo deja como riesgo abierto.
  • Escribe el plan QA en español dentro de la sub-tarea de Jira.
  • Crea el archivo .testcase.yml con todos los escenarios.
Entrega
Una sub-tarea de Jira con el plan listo + un archivo de test cases en el repo, trazable a la historia.
/qa-test-executor El Ejecutor
¿Qué hace? Toma el plan, abre el PR del desarrollador y corre la prueba real: antes vs. después del fix, regresión técnica, regresión de seguridad y validación funcional.
Paso a paso (en simple)
  • Lee el PR y clasifica el cambio (SQL, pipeline, Salesforce, UI).
  • Corre la prueba pre-fix y post-fix y compara resultados.
  • Revisa OWASP Top 10 y la regresión sobre lo que el cambio toca.
  • Levanta los defectos en Jira con evidencia adjunta.
Entrega
Un archivo .testrun.yml con PASS/FAIL, evidencia versionada y defectos creados con link a la historia.

Foco · Investigador de Cases (ya en producción)

🔍 Investigador de Cases

Implementación del agente Triage de Incidentes extendida al ciclo completo case → bug → fix → PR → ship, específica para el dominio Pod-Shop (Salesforce / POS). Divide el workflow en 3 skills encadenados con gates de validación humana entre fases, sobre los repos AuctiferaDX, POS y veevart-qa.
En producción
/case-triage Fases 1–4
¿Qué hace? Entiende el case, localiza el bug en código y confirma que es replicable antes de formalizar nada.
Paso a paso
  • Busca el case en Jira (Pod-Shop) y resume síntomas, ambiente y timeline.
  • Localiza clases sospechosas en AuctiferaDX, POS o veevart-qa.
  • Genera checklist de replicación con SOQL de verificación.
  • Clasifica complejidad y riesgo de regresión (🟢🟡🔴).
Gate: ¿Pudiste replicarlo?
/bug-fix Fases 4–5
¿Qué hace? Formaliza el bug en Jira y guía el fix con un checklist de testing completo.
Paso a paso
  • Crea el ticket (Bug u Hotfix) con causa raíz, componente y pasos.
  • Sugiere el approach de fix según el patrón detectado.
  • Checklist de testing: unit tests, regresión manual e integridad de datos.
Gate: ¿Pasaron todos los checks?
/case-ship Fases 6–7
¿Qué hace? Redacta el PR y corre el checklist de deploy para cerrar el ciclo limpiamente.
Paso a paso
  • Descripción del PR: summary, root cause, changes y test plan.
  • Checklist pre-deploy: CI verde, validación en UAT, cobertura ≥ 75%.
  • Verificación post-deploy, cierre del case y plan de rollback.
Cierra el ciclo ✓

/case-investigation Orquestador
Versión end-to-end que corre las 7 fases de corrido cuando no hace falta pausar entre skills. Mismo flujo, una sola invocación: del número de case al PR mergeado.
Relación con el mapa de agentes
Es la implementación concreta del agente Orquestador (todavía propuesto en el mapa de arriba) para el dominio Pod-Shop. Sirve como prueba de concepto del patrón orquestador antes de generalizarlo a otros pods.
Conectores Infra
Jira (veevart.atlassian.net) para cases y tickets. GitHub para código y PRs en los repos del pod.
Proyectos y repos en scope
  • Jira: Pod-Shop (SH), Phoenix-QA (PQ).
  • Repos: Veevarts/AuctiferaDX, Veevarts/POS, Veevarts/veevart-qa.
  • Contexto Salesforce del usuario para resolver org, sandbox y perfil.
